# 技術ブログ執筆チェックリスト

執筆・レビュー時に確認する。すべてを満たす必要はないが、意識的に判断する。

## 構成

- [ ] 記事の種類を1つに決めているか（チュートリアル / How-to / 概念解説 / リファレンス。混ぜない）
- [ ] 最初の2文で「何の記事か」「誰向けか」がわかるか
- [ ] TL;DRだけ読んでも価値があるか
- [ ] 見出しだけ読んで記事の流れが把握できるか
- [ ] 前提条件・スコープ（何を扱い、何を扱わないか）を明示しているか
- [ ] 結論や要点を先に述べているか（最後まで読まないとわからない構成になっていないか）
- [ ] セクション間の流れが自然か（唐突な話題転換がないか）

## 内容

- [ ] 実体験・知見に基づいているか（ドキュメントの要約になっていないか）
- [ ] 「なぜそうするのか」を書いているか（手順だけになっていないか）
- [ ] トレードオフや制限事項に正直か（良い面だけ書いていないか）
- [ ] 難しい部分に説明を厚く、簡単な部分は薄くしているか（逆になっていないか）
- [ ] 曖昧な情報に「未確認だが」「おそらく」等の留保をつけているか
- [ ] バージョン・日付を明記して情報の鮮度がわかるようにしているか

## コード

- [ ] コピーして動かせるか（依存関係・import漏れがないか）
- [ ] コードの前に「何をするコードか」を説明しているか
- [ ] 最小限の例から始めて、段階的に複雑にしているか
- [ ] 実行結果やoutputを示しているか（読者が正しく動いたか確認できるか）
- [ ] コード内コメントは「なぜ」だけに使い、「何」は本文で説明しているか
- [ ] 省略部分は `// ... (省略)` のように明示しているか（`...` だけにしない）
- [ ] CLIコマンドのプレースホルダーは `<value>` ではなく変数（`VAR="value"`）にしているか（コピーして使いやすいように）

## 文章

- [ ] 用語が記事内で統一されているか（同じ概念に複数の呼び方を使っていないか）
- [ ] 初出の専門用語・略語を説明またはリンクしているか
- [ ] 手順には番号付きリスト、順序のないものには箇条書きを使っているか
- [ ] 条件を先に書いているか（「Xの場合、Yする」であって「Yする、Xの場合」ではない）
- [ ] 「この記事では〜を説明します」のような冗長な導入を避けているか
